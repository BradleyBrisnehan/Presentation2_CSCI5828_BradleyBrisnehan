Links

With modules containing Attributes and Objects, DOORS then allows links among Objects within each module. This allows a Test Plan Object to be linked to a Design Object, the Design Object to be linked to a System Requirement Object, and a System Requirement Object to be linked to a user requirement. Links can be used to track relations in both directions (User Requirements to System Requirements, and vice versa). This functionality facilitates the following improvements in productivity:

  • Failure of Test Plan: If an Object in the Test Plan module fails, the designers can quickly identify the User Requirement which will not be met. This may result in a user review to re-evaluate reuirement, a review of the test to better assess the requirement, or a review of the design.
  • Over Testing: If an Object in the Test Plan module cannot be linked through a Design Object to a System Requirement Object, then that test is not necessary. This will eliminate the desire to exhaustively test functionality which has already been verified elsewhere.
  • Design Scope Creep: If an Object in the Design module cannot be linked to a System Requirement then the fucntionality is "nice to have" but will cost time and money without satisfying user needs.
  • Inadequate Design: If an Object in the System Requirement module cannot be linked to a Design Object then we will not meet that requirement. Re-evaluation of the design and User Requirements must be initiated.
  • Change of User Requirement: If an Object in the User Requirements module, the design team can quickly identify the System Requirement, Design, and Test Plan modules which must be updated in response.

The last example leads to the final functionality of DOORS: Change Tracking. DOORS offers more than a simple binary (changed or not changed) change tracker. This functionality creates the collaborative portion of requirements control.
