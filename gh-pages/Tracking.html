Change Tracking

In a collaborative environment, all users must be able to view the information of interest to them with only certain individuals being able to edit the content. DOORS defines the following edit modes:

  • Read-Only access allows the user to view a module, without the ability to edit content.
  • Exclusive access allows one user to edit a module, while all other users can only view it.
  • Shareable access allows multiple users to concurrently edit a module. Users cannot edit the same Attribute or Object at the same time.

A status bar at the bottom of the window indicates the edit mode currently employed. The user can change to any edit mode (which policies allow them to) for the given module.

DOORS maintains a "revision of record" style database. This means that when an Object is changed, DOORS will record information concerning when the change was made, the old object, and the new object. This allows users to track through changes over time to understand what requirements were previously and how they have evolved. In addition, DOORS defines baselines as a read-only module "snapshot." These can be used to mark a major milestone in the project life-cycle.

DOORS has the following change states:

  •  New Object: The Object has been created during the current session but has not been saved.
  •  Unsaved Changes: The Object has been edited during the current session but has not been saved.
  •  Baseline: The Object has not been edited since the last baseline.
  •  Modified and Saved Since Baseline The object has been modified and saved since the last baseline.
  •  Deleted The object has been deleted.

The change state is indicated by a change bar along with a tooltip providing additional detail. If more detail is required beyond the change bar and tooltip, the user can view the detailed history of the module. All of this functionality makes it very easy for users to collaborate and understand the evolution throughout the software life cycle.

Finally, individuals inside and outside of the organization can review the structure of requirements in DOORS and use a Change Proposal System to make requests. Each request must be reviewed by a Change Review Board to determine if further consideration and integration of the changes must be completed. Two proposals are possible in the system:

  • Suggestions are a high-level input for the structure of the requirements. This input requires the team to determine the details of the actual implementation.
  • Change Proposals are detailed inputs concerning a specific object. this input defines the specific change and must be discussed further if a different change is implemented.

All of these change methods allow for a dynamic level of collaboration among users, designers, managers and more.
