Modules

Once requirements have been defined, they must be organized in a logical / usable manner. DOORS organizes requirement information as a set of modules and links among them. This allows information to be tracked throughout project at different levels. Common modules within DOORS are:

  • User Requirements define the features and functionality that the user has requested. These requirements define the problem domain.
  • System Requirements define the requirements in a testable / verifiable manner. These requirements define the solution domain.
  • Design defines the structure of the final solution. The design should thoroughly satisfy System Requirements. Design features not linked to System Requirements lead to higher time and monetary costs.
  • Test Plans define the method for verifying that the Design satisfies the System Requirements.

Each module is then subdivided into Objects and Attributes. Attributes of the module is information such as Created By or Modified On. Objects are hierarchical tree which defines the detail of the module structure. Two basic columns of each Object is the heading and text - which is displayed to allow quick navigation through the Object tree. The user can define additional columns including priorty, time, monetary cost, among others.

The modules is a powerful tool to organize each phase of development. DOORS does not restrict the development within each module to any phase of the project. This allows it to support Waterfall development methods just as much as it supports Agile methods. As long as objects are properly defined within each module, the progression can always be traced through the phases of the project. This brings us to the next functionality in the DOORS suite - links among objects ease development among each phase.
